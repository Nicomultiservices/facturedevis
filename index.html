📧 Envoi automatique d'emails (EmailJS)

Connectez votre compte EmailJS (gratuit) pour envoyer automatiquement un email au client à chaque création de devis/facture, et un rappel de rendez-vous la veille à 18h. Vous recevrez systématiquement une copie.
Créez un compte sur emailjs.com, un "Service" (ex: Gmail), puis deux templates : un pour les devis/factures et un pour les rappels de RDV (vous pouvez utiliser le même). Récupérez ensuite les identifiants ci-dessous.

L'adresse de la page de consultation client se configure maintenant dans la carte 🔗 Liens de vos autres pages, un peu plus bas sur cette page.
⚠️ Configuration du template EmailJS (important) :
Sur emailjs.com, dans votre template, le champ "To Email" doit contenir exactement {{to_email}} (et non {{cc_email}} ni votre propre adresse en dur) — c'est ce qui determine le VRAI destinataire. Le champ "Cc" doit contenir {{cc_email}} pour recevoir une copie. Si "To Email" est mal rempli (ex: votre adresse au lieu de {{to_email}}), tous les mails partiront chez vous au lieu du client.

Variables a utiliser dans le corps du template : {{to_name}}, {{subject}}, {{message}}, {{from_name}}.
Pour la piece jointe PDF (devis/factures) : ajoutez un champ de type fichier joint nomme attachment dans les parametres du template EmailJS — le PDF reel sera automatiquement joint a l'envoi.

📥 Connexion Gmail (voir les emails reçus)

Contrairement à EmailJS (qui envoie uniquement), cette connexion vous permet de lire les emails reçus dans votre boîte Gmail, directement depuis l'onglet "Emails reçus". Lecture seule : l'application ne peut ni envoyer, ni supprimer, ni modifier vos emails Gmail.

Étapes de configuration (une seule fois) :
1. Allez sur console.cloud.google.com et créez un projet (ou utilisez un projet existant).
2. Menu "API et services" → "Bibliothèque" → activez Gmail API.
3. "API et services" → "Écran de consentement OAuth" → type Externe → renseignez un nom d'app et votre email, ajoutez votre propre adresse Gmail comme "utilisateur test".
4. "Identifiants" → "Créer des identifiants" → ID client OAuth → type "Application Web".
5. Dans "Origines JavaScript autorisées", ajoutez l'adresse exacte à laquelle vous ouvrez cette application (ex : https://votre-site.github.io ou http://localhost si testé en local).
6. Copiez le Client ID généré (il ressemble à xxxxx.apps.googleusercontent.com) et collez-le ci-dessous.

🛠️ Mode maintenance

Bloque immédiatement l'accès à votre site vitrine et à l'espace client (consultation.html) avec un message d'attente. Utile pendant une grosse mise à jour. gestion-devis-factures.html reste toujours accessible pour vous permettre de continuer à travailler.

Verrouillage ciblé : bloque uniquement l'espace client (consultation.html), sans toucher au site vitrine. Utile en cas de doute sur une faille de sécurité ou un accès suspect, en attendant d'investiguer.

🔧 Mode maintenance de l'application elle-même

Contrairement au mode maintenance ci-dessus, ceci bloque gestion-devis-factures.html lui-même : l'écran de connexion est remplacé par le message ci-dessous pour tout le monde, y compris vous. Un code de déblocage vous permet de repasser derrière et d'accéder normalement à l'écran de connexion.

⚠️ Notez bien votre code avant d'activer : sans lui, vous ne pourrez plus revenir sur cet écran pour désactiver la maintenance vous-même (il faudrait modifier le document Firestore siteConfig/main directement depuis la console Firebase).
🔍 Diagnostic : si l'écran de maintenance ne s'affiche pas alors que vous venez d'activer, ce navigateur a probablement déjà mémorisé un code de déblocage valide (d'une activation précédente). Utilisez le bouton ci-dessous pour l'oublier et forcer le test sur cet appareil.

📢 Bannière d'annonce globale

Un message informatif affiché en haut de l'application à tous les comptes connectés (ex : "Maintenance dimanche 22h", "Nouvelle fonctionnalité : ..."). Contrairement au mode maintenance, l'application reste entièrement utilisable — chaque utilisateur peut fermer la bannière de son côté.

💾 Quota de stockage par compte

Chaque compte stocke toute sa base dans un seul document Firestore, limité à 1 Mo. Au-delà, les sauvegardes échouent silencieusement pour ce compte. Surveillez ici les comptes qui approchent de la limite (beaucoup de devis/factures/photos scannées).

Cliquez sur « Charger » pour analyser les comptes.

✉️ Test d'envoi email

Envoie un email de test à votre propre adresse (celle de vos paramètres entreprise), pour vérifier qu'EmailJS fonctionne réellement sans avoir à créer un faux client ni un faux devis.

🧹 Purge des documents scannés anciens

Les documents scannés (photos/PDF) sont souvent la cause d'un dépassement de la limite de taille Firestore (1 Mo). Supprimez ceux qui datent de plus d'un certain délai pour libérer de l'espace — ils resteront disponibles en pièce jointe si vous les avez déjà envoyés par email.

🩺 Diagnostic de la configuration

Teste automatiquement Firebase, EmailJS et l'accès aux collections publiques (celles utilisées par consultation.html et vitrine.html), pour repérer un problème de configuration avant qu'un client ne le rencontre.

🗂️ Bases de données par compte

Chaque compte créé sur cette application dispose par défaut de sa propre base de données isolée (ses propres clients, devis, factures), bien que tous soient stockés sur le même projet Firebase. Depuis cette section, activez le partage pour que plusieurs comptes travaillent sur une base commune (ex : vous et un associé), ou laissez désactivé pour garder chaque compte totalement séparé (ex : vous louez l'application à d'autres artisans indépendants).

🟢 Partage activé : les comptes membres voient et modifient les mêmes clients/devis/factures.
🔴 Partage désactivé (par défaut) : chaque compte ne voit que ses propres données.
